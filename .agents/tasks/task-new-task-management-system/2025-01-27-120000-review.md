# Task management panel for ThoughtFlow

A complete task management system added as a sidebar panel (`TaskPanel.jsx`) with CRUD operations, section-based grouping (Today/Upcoming/No Due Date/Completed), search, status filtering, manual reordering, and canvas location-pin integration. The panel follows the existing pattern established by PinPanel and ReminderPanel — component in its own file, state held in App.jsx, persistence piggybacked onto the project/localStorage system. Tasks are project-scoped (shared across all workspaces within a project), while their location pins live inside individual workspaces.

**Watch for:** Reorder logic operates on a two-partition model (completed vs. everything else) while the UI shows four sections, so "Move Up" can swap a task across visible section boundaries (confirmed). Deleting a task orphans its location pin on the canvas with no cleanup (confirmed). Tasks are project-global while pins are workspace-local, creating an implicit cross-scope reference that becomes confusing in multi-workspace projects (confirmed).

## High-level view

The reorder mechanism in App.jsx swaps `sortOrder` values between adjacent tasks in a two-way partition (completed / not-completed). The TaskPanel renders four sections within the "not-completed" partition, so a reorder operation can move a task past a section boundary the user doesn't expect (e.g., swapping a "Today" task with an "Upcoming" task that happens to be adjacent by sortOrder).

Tasks live at the project level (same scope as reminders), not inside individual workspaces. Location pins, however, are per-workspace. A task's `locationPinId` references a pin that only exists in the workspace where it was created. If the user switches workspaces, "Go To Location" will silently fail because `navigateToPin` returns early when the pin isn't found in the active workspace.

`sortOrder` assignment uses `tasks.length`, which drifts after deletions and can produce duplicate values. The reorder swap logic assumes unique sortOrder, so collisions cause unpredictable ordering after several create/delete cycles.

<summary>Issues (5)</summary>

1. **Reorder crosses section boundaries** — `reorderTask` partitions only by completed/non-completed, but the UI has four sections. A "Move Up" on the first Upcoming task can swap it with the last Today task. Partition the reorder logic by the same four-section grouping the UI uses.
2. **Orphaned location pins on task delete** — `deleteTask` removes the task but leaves its associated pin on the canvas. Either delete the linked pin or null out the reference.
3. **Cross-workspace pin reference** — Tasks are project-scoped but `locationPinId` points to a pin in a specific workspace with no stored workspace ID. "Go To Location" fails silently in other workspaces. Store the workspace ID alongside the pin ID.
4. **Date.now() ID collisions on rapid creation** — Both `addTask` and the location-pin creation use `Date.now()` for IDs. Two rapid creations within the same millisecond produce duplicates. Append a random suffix or use a counter.
5. **sortOrder gaps after deletion** — Deleting tasks leaves holes in `sortOrder` values. After many create/delete cycles, new tasks (assigned `sortOrder = tasks.length`) can collide with existing values, breaking ordering. Compact sortOrder on delete or use a different assignment strategy.

## Reorder model vs. section model mismatch

The `reorderTask` function in App.jsx splits the task array into exactly two groups: completed and not-completed. It sorts within the group by `sortOrder`, finds the target task's index, and swaps `sortOrder` with the adjacent task.

```javascript
const isCompleted = task.status === 'completed';
const sectionTasks = prev.filter(t => (t.status === 'completed') === isCompleted)
  .sort((a, b) => (a.sortOrder || 0) - (b.sortOrder || 0));
```

The TaskPanel renders non-completed tasks in three separate sections: Today (due today or overdue), Upcoming (future due date), and No Due Date. Within each rendered section, "Move Up" / "Move Down" buttons invoke `onReorderTask`, but the swap happens against the entire non-completed pool ordered by sortOrder. A user clicking "Move Up" on the first item in "Upcoming" can swap sortOrder with a task that renders in a completely different section — a no-op visually, or worse, it scrambles ordering across sections.

The fix is to make the reorder logic aware of the same four-section grouping the UI uses (passing the section key or due-date criteria alongside taskId and direction), so swaps only happen within a single visual section.

## Location pin lifecycle gap

When a task is deleted via `deleteTask`, the function simply filters it out of the array:

```javascript
const deleteTask = (taskId) => {
  setTasks(prev => prev.filter(t => t.id !== taskId));
};
```

If that task had a `locationPinId`, the associated pin remains in the workspace's pin array indefinitely. Over time, a user who creates and deletes many tasks will accumulate "Task Location" pins (🎯 markers) on the canvas with no connection to anything.

The reverse is also true: if a user deletes a pin through the PinPanel, any task referencing that pin retains a stale `locationPinId`. The "Go To Location" button remains visible in the edit form but does nothing — a dead button that confuses rather than crashes.

## Tasks are project-global, pins are workspace-local

The `tasks` state array is stored at project level — a single flat list shared across all workspaces. Location pins live inside `workspace.pins[]`. When the location selection workflow fires, it creates a pin in the *currently active* workspace:

```javascript
updateActiveWorkspace(ws => ({
  pins: [...(ws.pins || []), newPin],
}));
setTasks(prev => prev.map(t => t.id === selectingLocationForTaskId ? { ...t, locationPinId: pinId } : t));
```

The task stores only `locationPinId` with no record of which workspace the pin lives in. If the user switches to a different workspace and opens the task panel, "Go To Location" calls `navigateToPin(pinId)` which searches `activeWs.pins` — a different workspace — and silently fails.

Either tasks should store `{ locationPinId, locationWorkspaceId }` so navigation can switch workspaces first, or the "Go To Location" button should be hidden/disabled when the linked pin isn't in the active workspace.

## sortOrder assignment drift

New tasks are assigned `sortOrder: tasks.length`. After several create/delete cycles, `tasks.length` decreases but existing tasks retain their old sortOrder values. Example: create 5 tasks (sortOrder 0-4), delete tasks at indices 2 and 3, create a new task — it gets sortOrder 3, which collides with an existing task. The swap logic in `reorderTask` assumes unique sortOrder values, so collisions produce unpredictable results when the user tries to reorder.

Fix: assign `sortOrder: Math.max(0, ...prev.map(t => t.sortOrder || 0)) + 1` to guarantee uniqueness, or compact all sortOrder values after deletion.
