# Card editor side panel for comfortable long-content editing

The change adds a dedicated `CardEditorPanel` component — a right-side panel that mirrors the selected card's title and content into full-size input fields with a live markdown preview below. It follows the established side-panel pattern (w-80, border-l, bg-white, shrink-0) and wires into the existing `updateNode` / `selectedNodeIds` infrastructure. The panel only renders in canvas view and only shows editing fields when exactly one node is selected.

**Watch for:** The `onUpdateNode` handler in App.jsx calls `takeSnapshot()` on every keystroke (confirmed), which will flood the undo history with per-character entries and make Ctrl+Z unusable for meaningful rollback after panel edits. The `useEffect` sync has `selectedNode?.title` and `selectedNode?.content` in its dependency array, which creates a redundant state-set cycle on each change (likely benign but wasteful).

## High-level view

The panel uses local `title`/`content` state synced from the selected node via `useEffect`, with change handlers that write-through to the canvas node immediately.

The undo integration is the one behavioral mistake. Every other editing path in the app calls `takeSnapshot()` once when the user enters the editing field (on focus or first click), then updates freely until the editing session ends. The panel instead snapshots on every change event, meaning typing "hello" generates five undo entries instead of one. This makes undo behavior inconsistent with canvas editing and will surprise users.

The `useEffect` dependency on `selectedNode?.title` and `selectedNode?.content` means the effect fires every time the parent re-renders with updated node data — including re-renders triggered by the panel's own changes. Since `setTitle` with the same value is a no-op in React, this doesn't cause visible bugs, but it's architecturally fragile. If the panel ever introduces any transformation (trim, truncation), the feedback loop would oscillate.

## Undo history flooding via takeSnapshot per keystroke

In App.jsx, the `onUpdateNode` callback passed to the panel is:

```jsx
onUpdateNode={(updates) => {
  if (cardEditorNode) {
    takeSnapshot();
    updateNode(cardEditorNode.id, updates);
  }
}}
```

Every call to `handleTitleChange` or `handleContentChange` — i.e., every keystroke — triggers this. Compare with the canvas inline editing pattern:

```jsx
// Canvas title editing
onFocus={() => takeSnapshot()}
onChange={(e) => updateNode(node.id, { title: e.target.value })}
```

The canvas takes one snapshot when the user starts editing. The panel takes one per character. For a 100-character edit, the undo stack gains 100 deep-clone snapshots of the entire workspace state (`JSON.parse(JSON.stringify(stateRef.current))`). This is both a UX and a memory concern.

The fix is to move `takeSnapshot()` out of the per-change callback. Options: call it once when `cardEditorNode` first becomes non-null (i.e., when user selects a card with panel open), or add an `onFocus` handler to the input/textarea that takes a snapshot before the first change, matching the canvas pattern exactly.

## useEffect sync creating a render feedback loop

```jsx
useEffect(() => {
  if (selectedNode) {
    setTitle(selectedNode.title || '');
    setContent(selectedNode.content || '');
  } else {
    setTitle('');
    setContent('');
  }
}, [selectedNode?.id, selectedNode?.title, selectedNode?.content]);
```

The dependency array includes `selectedNode?.title` and `selectedNode?.content`. When the user types in the panel:

1. `handleTitleChange` calls `setTitle(newValue)` + `onUpdateNode({title: newValue})`
2. `onUpdateNode` calls `updateNode`, which calls `setWorkspaces(...)`
3. React re-renders App → new `cardEditorNode` object with updated `title`
4. `selectedNode.title` changes → useEffect fires → `setTitle(selectedNode.title)` (same value, no-op)

The cleaner pattern: depend only on `selectedNode?.id` and reset local state only when the selected node changes. While the user is actively editing, local state is authoritative; prop state is just the initial value. This also eliminates the fragility where any future transformation in the change handler (trim, truncation, sanitization) would cause oscillation.
