# View/edit separation for TaskPanel and FullTaskManager

The change splits click interactions in both task views into a two-tier model: single click selects a task and reveals a read-only notes preview, double click (or an explicit Edit button) opens the full edit form. The sidebar grows from w-80 to w-96, row padding shrinks, and the full-screen table header is rebalanced with tighter columns and a new Group badge column. The intent is to let users browse tasks without accidentally triggering edit mode.

Watch for: TaskPanel's single click does not toggle selection off (confirmed), creating a "sticky" selection with no escape hatch short of double-clicking another task or editing. FullTaskManager's `handleRowDoubleClick` reuses the name but fires a different code path than actual double-click — when called from the Edit button, the surrounding single-click handler fires first, setting `selectedTaskId` before `handleRowDoubleClick` clears it (likely).

## High-level view

The sidebar separates `selectedTaskId` and `editingTaskId` into two distinct states but implements `handleTaskClick` as a pure setter rather than a toggle — clicking an already-selected task does not deselect it. The FullTaskManager toggle works correctly but the inconsistency between the two views may confuse users who switch between them.

The FullTaskManager column rebalancing moves Location from a prominent 12-unit column to a compact 8-unit trailing position, inserts a Group badge column, and consolidates reorder arrows with a new Edit button into a single Actions column. The column order now differs significantly from the sidebar's row layout, and the Group badge styling diverges between views (`text-[9px] rounded-full bg-slate-100` in full-screen vs plain `text-[10px] text-slate-400` text in the sidebar).

1. **TaskPanel click does not deselect** — `handleTaskClick` always sets `selectedTaskId` to the clicked task, never `null`. Users cannot collapse the notes preview by clicking the same row again. Change to a toggle: `setSelectedTaskId(prev => prev === task.id ? null : task.id)`.

3. **No keyboard escape from selection** — Neither component listens for Escape to clear `selectedTaskId`. The edit form handles Escape (in TaskPanel via `onKeyDown`), but the lightweight selection state has no keyboard dismiss path.

## Selection toggle asymmetry between views

In FullTaskManager, `handleRowClick` toggles: clicking an already-selected task sets `selectedTaskId` to `null`. In TaskPanel, `handleTaskClick` always assigns:

```jsx
// TaskPanel — no deselect path
const handleTaskClick = (task) => {
  setSelectedTaskId(task.id);
  setStatusDropdownTaskId(null);
};

// FullTaskManager — toggles correctly
const handleRowClick = (task) => {
  if (selectedTaskId === task.id) {
    setSelectedTaskId(null);
  } else {
    setSelectedTaskId(task.id);
  }
};
```

A sidebar user who single-clicks a task to see notes has no way to dismiss the preview except by double-clicking (which enters edit mode) or selecting a different task. The fix is one line — add the same toggle guard to `handleTaskClick`.

## Edit button event propagation in FullTaskManager

The Edit button in the Actions column calls `e.stopPropagation()` before invoking `handleRowDoubleClick(task)`. `handleRowDoubleClick` calls `setSelectedTaskId(null)` internally, while the row's `onClick` (if it somehow fired first) would set `selectedTaskId` to the task. Because `stopPropagation` is called synchronously before the row handler can execute, the race is unlikely — but a quick manual click test would confirm the notes preview never flashes between selection and edit states.

## Column layout divergence

The full-screen table now uses this column order: Status | Task Name | Group | Priority | Due | Location | Actions. The sidebar uses: Status + Title (Row 1), Group + Priority + Due (Row 2). The Group badge in full-screen uses `text-[9px] ... rounded-full bg-slate-100` while the sidebar uses plain `text-[10px] text-slate-400 font-medium` text. If users mentally map between the two views, the inconsistent visual treatment of the same data may cause friction.
