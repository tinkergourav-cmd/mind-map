# Task-Pin Location system replacing Card-Task links

The change removes the `cardTaskLinks` state array (which mapped tasks to card IDs) and replaces it with an inline `locationPin: { x, y, canvasId }` field on each task object. Tasks now store canvas coordinates directly, breaking the dependency on card existence. A "Drop Task Location" interaction mode lets users click the canvas to place or update a pin, and task location pins are rendered as draggable üéØ markers with hover tooltips and bounce-on-navigate animation. Migration paths handle old `cardTaskLinks` data on both project load and single-file import.

Watch for: a **confirmed** runtime crash in `importAllData` where the removed `setCardTaskLinks` is still called; a **confirmed** incorrect `canvasId` assignment during migration that puts all pins on `activeTab` regardless of which workspace contains the card; and a **likely** UX regression where repeated "Add to Tasks" from the same card now creates duplicate tasks with no guard.

## High-level view

The migration paths (project load and single-file import) find the linked card's coordinates and stamp them onto tasks, but both use `activeProj.activeTab` as the `canvasId` for every migrated task rather than resolving which workspace actually contains the card. For multi-tab projects with cross-tab card links, this places the pin on the wrong canvas.

The `importAllData` function (full-backup restore) still calls `setCardTaskLinks`, which no longer exists. This will throw a `ReferenceError` at runtime when a user imports a full backup, causing the entire restore operation to fall into the catch block and show an error message while leaving the app in a partially-loaded state.

The old duplicate-task guard (`cardTaskLinks.some(l => l.cardId === cardId)`) was removed without replacement. Users who right-click a card and select "Add to Tasks" multiple times (or press Shift+T repeatedly) will now create duplicate tasks for the same card, each with its own location pin at the same coordinates.

## `importAllData` still references removed state

At line 2283, inside the full-backup import path:

```javascript
setCardTaskLinks(defaultProj.cardTaskLinks || []);
```

This identifier no longer exists. The call sits inside a try/catch, so when it throws, the catch block fires `setErrorMessage("Import failed...")` and rolls back to `previousProjects`. The user sees an error; the backup is not restored. This is a data-loss path for users upgrading from old backup files. The fix is to either apply the same migration logic used in project-load, or simply remove the line and let the tasks load as-is (they'll have no `locationPin` until the user sets one manually).

## Migration assigns all pins to `activeTab`

Both the project-load migration (line 711) and the import migration (line 2215) resolve the card's `x` and `y` from `initialWorkspaceNodes` (which correctly flattens all workspaces), but then assign `canvasId: activeProj.activeTab || ''`. If a task was linked to a card on a non-active tab, the migrated pin points to coordinates that may not correspond to any content on the active tab.

The fix is straightforward: when iterating workspaces to find the card, capture the workspace ID:

```javascript
let foundCanvasId = '';
for (const ws of initialWorkspaces) {
  const card = (ws.nodes || []).find(n => n.id === link.cardId);
  if (card) {
    foundCanvasId = ws.id;
    // use card.x, card.y, foundCanvasId
    break;
  }
}
```

## Duplicate task guard removed

The old code prevented creating multiple tasks for the same card via `cardTaskLinks.some(l => l.cardId === cardId)`. The new code has no equivalent guard. Since tasks now store coordinates (not card IDs), there's no natural dedup key. If this is intentional‚Äîallowing multiple tasks at the same location‚Äîit should at minimum avoid identical titles piling up. If unintentional, a guard like `tasks.some(t => t.title === card.title && t.locationPin?.x === card.x && t.locationPin?.y === card.y)` would approximate the old behavior.

## Drop mode lacks keyboard escape

The `droppingTaskLocation` mode changes the cursor to crosshair and shows a banner. The only way to cancel is the banner's Cancel button. The existing Escape-key handler (line 1360) only clears `selectedNodeIds`. Adding `if (droppingTaskLocation) { setDroppingTaskLocation(null); return; }` to that handler (or a new one) gives users the standard escape hatch they expect from a modal interaction mode.

## Cross-tab navigation timing

`locateTaskPin` reads `workspaceRef.current.getBoundingClientRect()` and then conditionally calls `setActiveTab(canvasId)`. If the tab switch triggers a re-render that changes the canvas container's dimensions (e.g., different content causing different overflow), the rect captured before the switch may produce off-center positioning. In practice this is likely unnoticeable for same-dimension layouts, but if tabs ever have different panel configurations, the centering will drift.
