# Task-Pin Location system replacing Card-Task links (v2)

The change removes the `cardTaskLinks` state and replaces it with an inline `locationPin: { x, y, canvasId }` field on each task. Tasks store canvas coordinates directly, breaking the dependency on card existence. A "Drop Task Location" interaction mode lets users click the canvas to place or update a pin, and task pins render as draggable đźŽŻ markers with hover tooltips and bounce-on-navigate animation. Migration paths handle old `cardTaskLinks` data on project load, single-file import, and full-backup restore.

Watch for: a **likely** migration gap in the single-file import path where malformed `tasks` arrays can cause migration to run against the wrong task set; and a **possible** single-frame timing edge on cross-tab navigation if the workspace container re-layouts asynchronously.

**Verdict**: APPROVED

## High-level view

All five issues from the v1 review have been addressed. The `setCardTaskLinks` crash is replaced with migration logic that iterates workspaces to resolve the correct `canvasId`. The duplicate guard uses coordinates-plus-title equality. Escape cancels drop mode. Cross-tab centering defers via `requestAnimationFrame`.

One sequencing gap remains in the single-file import: the `cardTaskLinks` migration block runs unconditionally, even if the preceding task validation failed and `setTasks` was never called with the imported data. This can apply migration against stale tasks from the previous project.

The duplicate guard in `addTaskFromCard` compares exact coordinates and title, so moving a card after initial task creation allows a second task for the same card. This is consistent with the new architecture â€” the pin is a location, not a card reference â€” but represents a behavioral difference from the old ID-based dedup.

## Migration sequencing in single-file import

The single-file import path at line ~2209 first validates and sets tasks:

```javascript
if (Array.isArray(importedData.tasks) && importedData.tasks.every(t => t && t.id && t.title != null)) {
  setTasks(importedData.tasks);
}
// Migration always runs if cardTaskLinks exist, regardless of whether tasks were set above
if (Array.isArray(importedData.cardTaskLinks) && importedData.cardTaskLinks.length > 0) {
  setTasks(prev => prev.map(t => { ... }));
}
```

If the tasks array has a single malformed entry (one task without a `title`), the `every()` check fails and `setTasks(importedData.tasks)` is skipped. The migration then runs against whatever tasks were in state before the import â€” applying `cardTaskLinks` from the imported file to tasks from the current project. The result is either no-ops (IDs don't match) or incorrect pin placement on tasks that happen to share IDs.

Fix: nest the migration block inside the task validation `if`, or track a `tasksImported` flag.

## Cross-tab centering timing

The workspace container uses `transition-all duration-300`. If a tab switch coincides with an ongoing panel resize animation, `getBoundingClientRect()` at rAF time captures mid-transition dimensions, producing slightly off-center positioning. This is cosmetic and self-correcting on the next navigation, but a two-frame deferral (`requestAnimationFrame(() => requestAnimationFrame(centerOnPin))`) would eliminate it.
